The microphone we used to record was sort of finicky, and would pick up on a lot of unwanted noise. Also, the way it chose sound to be detected was slightly problematic: for one, it would often be unable to really discriminate between noise vs real audio; additionally, it would sort of “balance out” volume as it recorded, resulting in noise being amplified (especially during moments where there was supposed to be silence); both of these issues resulted in really loud noise throughout recordings, particularly in the “silent” moments before and after the actual audio (the user’s delay in reaction time between pressing/releasing the record button and actually making the noise to be recorded). We tried to address this problem by having “noise thresholds”, and when processing the audio, remove data from the beginning and end of the recording that fell underneath these thresholds. These were effective in removing noise from certain types of sounds (in particular loud sounds, continuous sounds, sounds with distinct/sharp beginnings like a tapping something that resonates), but were harmful for other types of sounds (in particular softer noises, less resonant sounds, speech, etc.) and removed a lot of needed samples, trimming the samples to be way too short. The noise thresholds have been tuned to try to minimize losses while maximize noise removal, but it’s still pretty finicky (leans on the side of removing too much); for now, this is the best we can do using an amplitude-based noise processing method rather than doing advanced stuff like signal processing or analyzing frequencies. ## Memory A big issue we had was being able to store all of the audio data on the ESP32 in order to play the notes quickly. Because we were doing pitch shifting on the server and all audio files were stored on the server, we had to make a GET request each time a certain note was to be played. The decision to do so was because there were existing Python resources for pitch shifting, and working with changing audio data was more intuitive in Python. We tried to move pitch shifting on the ESP32, but that proved to be difficult, as attempted code would lead to ESP crashes and errors possibly due to memory issues. We also could not find a way to store all eight audio files onto the ESP32, so playing a single note has a slight lag from pressing the key to the sound actually playing. In general, there was an issue with the limited memory. Both the metronome and keyboard had functions that required the use of large arrays of data. At one point our code took up too much memory and would not compile, so we had to go through all of our arrays and see which ones could be shared and used for multiple purposes. ## Audio Screeching We also had audio file issues and playback issues, where oftentimes the audio would produce a high-pitched screeching sound due to a lack of audio. Although we tried reducing the screeching noise by only playing the length of the audio clip, there are still some screeching noises. Additionally, sometimes there is a bug where button presses would cause no sound to be played back and the ESP32 to register that an audio sample of length 0 was received from the server. We’re not sure the reason for this, but some guesses would be issues with manipulating memory/pointers to decode and store the audio data coming from the server, or perhaps the ESP32 loop speed causing issues in the proper execution sequence of certain code/functions (could also be caused by improperly-managed states in the state machine). ## Discretion with Arduino loop runtime The context of this design challenge was when I was trying to make the arduino make microphone analog reads at a high frequency (at first maybe 8kHz, later dropped to 1kHz), constant rate. For most projects and labs, we could simply assume that the arduino was able to run many loops over the main loop function every ms, but such assumptions break down when you’re coding a more involved project, with lots of “parallel” and time-consuming operations. I thought that tweaking previous code for sampling audio would directly give me what I wanted. I just had to change up the bit depth and sampling rate a bit. It turns out the previous keyboard recording already used a low bit depth of 8 bits, so further decreasing depth was probably not a good idea. I left the number of bins at 256. Since the recorded data did not need to be modulated or pitch shifted, there was really no reason to sample at a depth of 8kHz or anything crazy like that, so I went with 1kHz. To plot beats, I decided to store times when a beat occurred. Since I was only planning to record 10s intervals, I only needed an array of at most fifty 16 bit ints. All this was easy. The main difficulty with this task is integrating the beats and recorded audio into the same graph, making sure the x-axis labels for both are consistent! It seemed like this was no problem since I was making an analog reading once every ms (or so I thought) and knew at which ms the beats occurred at. It turns out the previous code for keyboard recording didn’t necessarily ensure that the analog reads happened at periodic intervals. I discovered this when trying to record a 10s stretch of audio realized the recording lasted maybe 15-20s instead. Partially due to limitations of the ESP32’s computational abilities and the bulk that was the rest of our code, there were some clock cycles that took more than 1 ms to run. I confirmed this by printing millis() every time I checked if a reading needed to be taken, and here were the readings I got (assume the recording starts at millis() = 0): $$0, 0, 0, 1, 1, 2, 2, 2, 3, 3, 7, 7, 8, 8, 9, 9, 10, 10, 10, 14,\ldots$$ The culprit behind this was our update_needle() function. Because we had added a metronome box to contain the needle in, we had to perform some expensive tft.draw_shape operations to make sure the needle’s movement didn’t utterly wreck the aesthetic of the box. These tft draws were causing our clock cycles to lag. I eventually decided to draw the metronome box only when not recording the metronome beats, so that now only the needle shows up when one tries to record if they are on beat or not. After this, analog reads synced will millis() to perform data inputs regularly, and all was good. # Code / File descriptions * Keyboard Metronome (.ino) * Display Func * The display function prints onto the LCD a screen associated with each different state in the machine. These statements tell the user how to use the device at each state, such as which button to press and what a press would do. It has its own state machine so that the main one remains easy to read and so that the screen is only updated when needed (no flickering). * Keyboard Func * This file houses the main functionality for the keyboard, where the notes are linked to their corresponding keys and the code detects whether or not keys are being played. If played, the note and octave variable will be updated and, using this information, a GET request is made to the server to retrieve the corresponding audio. Updating variables prevented us from having to write out the code for each possible GET request. This file also contains the code for the tilting detection, octave change, and scrolling keyboard display as well as the functions for removing audios and keeping track of the current audio ID. Removing an audio sends a request to the server based on the current audio ID and deletes the whole set of modulated audio files for that ID, otherwise, only some keys would be playable. After adding in the remove functionality, the existing IDs at a given point would not necessarily be in a pattern such as just increasing by 1 (1,2,3,4). As a result, we changed the way to keep track of IDs from just an ID variable that would increment by one to storing an updating array of existing IDs, and then keeping track of the index instead. This allows us to have an efficient way to display the correct ID, rather than keeping track of a growing list of ID numbers to skip. * KeyboardMetronome * The Keyboard Metronome file hosts our main state machine, that manages the keyboard, saved music, as well as the metronome. Through various inputs from the 8 keyboard buttons and one main button, this file transitions from state to state and calls necessary helper functions. Additionally, this houses all the necessary setup for the system, such as connecting buttons and LEDs, defining variables, and importing libraries. * Metronome Record Beat * This code file holds the functionality that allows one to record a 10 second stretch of audio and plot it in a bokeh plot on our webpage. Contains a setup() function to prepare to record beat, where it initializes a bunch arrays and counters to 0, timers to millis(), etc. The record_audio() function records one audio reading every ms and adds it to an array, calling the send_audio() function once it is finished. The send_audio() function delivers all the audio readings to the online server. Because the audio readings are 8 bit unsigned ints that are mostly in the range (50, 200), we don’t need to worry about encodings. However, base64 was used to send beat data, as the times at which beats occurred are 16 bit ints. * Metronome Stuff * The Metronome Stuff file had our main metronome state machine. As there were many different components to the metronome, we felt it made more sense to create an entirely separate state machine rather than nesting each metronome component into the main state machine. Thus, this file contains states to turn on the metronome, record tempo with clapping or buttons, and record and plot a sound to check how on-beat a user is. Additionally, there are functions to draw the metronome box and turn the needle red if off-beat or green if on-beat. * Support functions * Support_functions.ino contains your classic collection of http helper functions, like char_append() and do_http_request(). Additionally, we slapped all of thee copied base64 code from lab06a here and added a few more methods to create specialized GET and POST requests for sending and encoding/decoding base64 converted data. * Button * The button files creates a button class with various instances that help account for button presses, which we use for all our functionality. We have one function, button.update( ), which returns to the user a 1 if they made a short press, 2 if the press was held longer than 1 second, and 0 otherwise. We use the differences between long and short presses to extract more functionality from a limited number of buttons. * Audio_handler.py * This file is the main file that handles the recorded audio clips, interacting with the database and the client in order to insert newly recorded clips, return the selected audio data, and return a list of all current audio ID’s in the database. On the appropriate GET request, the user is able to request a specific audio clip, note, and octave, in which this file will then search the database and return the audio file data as base-64 encoded ASCII characters. On another GET request, the file returns a list of all ID’s in the database in order. Its functionality is for the audio selector state on the ESP32. There is also a POST request to deal with the ESP’s deletion of an audio file (more on deletion on server.py, both deletions are the same, but this one handle’s deletion from ESP, while server handles deletion from the webpage). Finally, the POST request to this file containing the audio data is stored into the databases. It also modulates the audio clip into 2 octaves of notes to be stored as well using pitchshifter.py. This is also where the audio file is converted into a WAV file and saved as well. * Last_id_index.py * This file contains a database that stores the most recently used indices of the array populated with existing audio ID’s. Through a GET request, this file will either insert a newly last-used index or return the most recent entry to the ESP when it is powering on. This allows the selected audio to last through power cycles. * Pitchshifter.py * This file contains a number of functions that takes a Python array of 8-bit integer audio data and a specified number of semitones to modulate this sample. It converts the array into a NumPy array and modulates it by the specified amount by stretching the length + changing the speed of the sample in the appropriate ratios. This process returns a NumPy that is converted back into a Python array of 8-bit integers; this is the modulated sample. * Record_beat.py * The code here has two parts: the first a database to store audio data and beat times, and the second a bunch of binary data manipulation. The binary_to_array() function takes an array of raw binary data, converts it to ints, and sends it to the database. We chose to make the database have two tables, one for beat times and one for binary data. The database lookup operations return the given data, which is used by bokeh to plot the data in a web browser, with the audio data as a line graph and beat times as red diamonds, evenly spaced along the y=128 line. * Server.py * This file houses the metronome database and its interactions, deletion of audio files, and deals with the interactions from the webpage. On a POST request (which is created by the webpage itself), it stores the tempo and beat on the database, for which the ESP32 can lookup using the appropriate GET request. The deletion of audio files occurs on its appropriate POST request, which deletes all audio clips of a specified audio ID in both the audio samples database and the WAV files. This file is also where the HTML of the WAV files are generated in order to allow the selecting and viewing of one audio file at a time, the deletion of audio, and a link to the audio’s plot. Regardless of the HTTP request, this file returns the HTML from index.html along with the additional generated audio HTML and its relevant CSS. * Index.html, style.css, and script.js files * These three files enable server.py to run the way it does and provides the styling and functionality of the nice sliders for metronome beat and tempo adjustment.
